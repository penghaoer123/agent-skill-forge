# 9 原则评分清单

> 每项 1-5 分。1=明显不符合，3=部分符合，5=高度符合。

## P1 · Description as Trigger（description 是触发器）

| 分 | 标准 |
|----|------|
| 1 | description 只是功能名（如"PDF处理"） |
| 2 | 描述了功能但无触发词 |
| 3 | 有触发词但不够全 |
| 4 | 触发词覆盖主要场景，缺反触发 |
| 5 | pushy：触发词 + 反触发 + 场景全覆盖 |

## P2 · Progressive Disclosure（渐进披露）

| 分 | 标准 |
|----|------|
| 1 | 所有内容塞正文，无 references |
| 2 | 有部分拆分但正文仍过重 |
| 3 | 分层存在但入口信息密度过高 |
| 4 | 三层清楚，入口简洁 |
| 5 | 正文一键可扫 + references 按需加载 + 索引完整 |

## P3 · 500-line Cap（500 行上限）

| 分 | 标准 |
|----|------|
| 1 | >500 行 |
| 3 | 450-500 行 |
| 5 | <450 行 |

## P4 · Scripts as Black Boxes（脚本黑盒）

| 分 | 标准 |
|----|------|
| 1 | 复杂流程内嵌 shell 片段，无脚本 |
| 2 | 有代码示例但未封装 |
| 3 | 部分流程脚本化 |
| 4 | 主要流程有 black-box 脚本但 SKILL.md 仍暴露细节 |
| 5 | 脚本 `--help` 自说明 + 正文只写调用方式 |

## P5 · Self-contained Folder（自包含）

| 分 | 标准 |
|----|------|
| 1 | 依赖大量外部路径、全局文件 |
| 2 | 有目录但依赖散落 |
| 3 | 基本自包含，少量外部引用 |
| 4 | 目录完整，外部引用有明确文档 |
| 5 | 完全自包含 + LICENSE + 所有依赖在目录内 |

## P6 · Opinionated over Generic（有判断力）

| 分 | 标准 |
|----|------|
| 1 | 纯模板，无判断 |
| 2 | 有少量偏好但不成体系 |
| 3 | 有明确偏好 |
| 4 | 有主张且可自圆其说 |
| 5 | 强烈主张 + 拒绝默认 + 反模式明确 |

## P7 · Iterate, Don't One-shot（迭代闭环）

| 分 | 标准 |
|----|------|
| 1 | 无验证/复盘机制 |
| 2 | 有 changelog 但无结构化迭代 |
| 3 | 有迭代记录但无显式 eval 流程 |
| 4 | 有 eval/验证 + 改进记录 |
| 5 | draft→eval→review→improve→repeat 完整循环 |

## P8 · Decision Tree Entry（决策树入口）

| 分 | 标准 |
|----|------|
| 1 | 线性流程，无分支判断 |
| 2 | 有分支但不在一屏 |
| 3 | 第一屏有决策树但覆盖不全 |
| 4 | 覆盖主要场景，缺默认分支 |
| 5 | 覆盖全场景 + "不确定"默认分支 |

## P9 · Ground in Subject（领域接地气）

| 分 | 标准 |
|----|------|
| 1 | 泛泛而谈，无具体领域知识 |
| 2 | 有领域但浅 |
| 3 | 有真实案例和坑点 |
| 4 | 有丰富的实践经验和教训 |
| 5 | 从真实使用、失败、修复、迁移中长出来的深度知识 |

## P10-P12 · v1.2 扩展原则

### P10 · Blocking Invariant Awareness（硬规则意识）

| 分 | 标准 |
|----|------|
| 1 | 技能内含 MUST 语句涉及审美/风格/流程习惯 |
| 2 | 有硬规则但散落各处 |
| 3 | 硬规则只在真正不可协商的点（安全/数据/真实） |
| 4 | 硬规则明确且不膨胀，反规则清晰 |
| 5 | 系统级 invariant 已由 coordinator 兜底，技能内无冗余 MUST |

### P11 · Gate Design（预检门设计）

| 分 | 标准 |
|----|------|
| 1 | 无任何执行前检查 |
| 2 | 有建议性检查但无失败处理 |
| 3 | 有关键依赖检查 + 失败处理 |
| 4 | 条件化 gate（只在相关任务触发） |
| 5 | gate 由 coordinator 路由触发，技能内只标注依赖 |

### P12 · Concurrency Awareness（并行感知）

| 分 | 标准 |
|----|------|
| 1 | 所有步骤按顺序写，无并行标注 |
| 2 | 有少量"可同时"提示 |
| 3 | 独立步骤有明确并行标注 |
| 4 | 并行/串行依赖清晰标注 |
| 5 | ∥ 和 → 标注 + coordinator 可据此并行调度 |
